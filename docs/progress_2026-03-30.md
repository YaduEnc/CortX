# SecondMind / CortX Progress Report (Updated March 31, 2026)

## Current Architecture (Active)

Audio path:

`ESP32 -> BLE -> iOS app -> backend websocket -> object storage`

The phone is now the uplink gateway. ESP no longer uploads audio directly to backend.

## Backend Status

Active APIs:
- `POST /v1/app/register`
- `POST /v1/app/auth`
- `GET /v1/app/devices`
- `POST /v1/pairing/start`
- `POST /v1/device/pairing/complete`
- `POST /v1/app/live/start`
- `GET /v1/stream/ws?stream_token=...` (WebSocket)
- `GET /v1/app/captures`
- `GET /v1/app/captures/{session_id}/audio`
- `GET /v1/app/captures/{session_id}/transcript`

Removed from active router:
- `/v1/capture/*` ingestion endpoints
- `/v1/stream/start` (device-start route)

## iOS App Status

Implemented:
- BLE pairing flow
- Live gateway flow (Start Live / Stop Live)
- BLE packet receive + sequence tracking + drop counter
- WebSocket uplink from app to backend
- Capture list + WAV playback

## Firmware Status

Implemented:
- BLE pairing characteristics
- BLE audio characteristics:
  - control (`start` / `stop`)
  - audio binary notify
  - audio state notify
- Continuous I2S capture -> BLE notify packets

Removed:
- Direct ESP websocket streaming/upload path
- Old chunk-upload/finalize logic in active Arduino sketch

## Documentation Status

Current source of truth:
- `docs/ble_phone_gateway_flow.md`
- `docs/iot_pairing_guide.md`
- `docs/app_pairing_api_flow.md`

Historical docs that referenced old routes were updated or removed where needed.
